# 04-测试用例与验收脚本清单

## 1. 目标
为 `UnityPilot MCP` 提供可执行的 MVP 测试清单，覆盖连接、编译、自动修复、PlayMode、鼠标模拟五大能力。

## 2. 测试范围
| 测试域 | 覆盖模块 | 目标 |
| --- | --- | --- |
| 连接与协议 | M01, M02 | 保证 WS 会话稳定、消息格式一致 |
| 编译与报错 | M03 | 保证编译触发和错误结构化输出可用 |
| 自动修复循环 | M04, M06 | 保证闭环可终止、可收敛、可观测 |
| PlayMode 控制 | M05 | 保证运行/停止可控 |
| 鼠标输入模拟 | M05 | 保证点击/拖拽等动作可执行 |

## 3. 测试前置条件
1. Unity 2022.3 LTS 已安装。
2. Python Orchestrator 已启动，监听 `127.0.0.1:8765`。
3. Unity 工程已导入 Bridge 插件并可自动连接。
4. 测试工程含至少 1 个可编译脚本和 1 个故意错误脚本。
5. 操作系统：Windows / macOS 各执行一轮。

## 4. 用例清单（MVP）
## 4.1 连接与协议
| 用例ID | 前置条件 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| WS-001 | Orchestrator 运行中 | 打开 Unity | 10 秒内收到 `session.hello`，会话状态为 connected |
| WS-002 | 已连接 | 持续观察 30 秒 | 每 2 秒有心跳消息，无断连 |
| WS-003 | 已连接 | 停止 Orchestrator 5 秒后再启动 | Unity 自动重连成功 |
| WS-004 | 已连接 | 发送未知命令 | 返回 `COMMAND_NOT_FOUND`，服务不中断 |

## 4.2 编译与错误收集
| 用例ID | 前置条件 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| CP-001 | Unity 已连接且空闲 | 调用 `unity.compile` | 返回 accepted 与 `compileRequestId` |
| CP-002 | 编译中 | 再次调用 `unity.compile` | 返回 `EDITOR_BUSY` |
| CP-003 | 工程含语法错误 | 调用 `unity.compile` 后读取 `unity.compile_errors` | 返回 `file/line/column/message/severity` |
| CP-004 | 修复语法错误后 | 再次编译并读取错误 | `errors=[]` 且 `total=0` |

## 4.3 自动修复循环
| 用例ID | 前置条件 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| FX-001 | 工程含 1 个可修复错误 | 调用 `unity.auto_fix_start(maxIterations=20)` | 循环最终 `success` |
| FX-002 | 工程含无法自动修复错误 | 调用 `unity.auto_fix_start(maxIterations=1)` | 循环 `max_reached` |
| FX-003 | 循环运行中 | 调用 `unity.auto_fix_stop` | 循环在下一轮前停止，状态 `failed` |
| FX-004 | 无编译错误 | 调用 `unity.auto_fix_start` | 1 轮内返回 `success` |

## 4.4 PlayMode 控制
| 用例ID | 前置条件 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| PM-001 | Unity EditMode | 调用 `unity.playmode_start` | 3 秒内进入 PlayMode |
| PM-002 | Unity PlayMode | 调用 `unity.playmode_stop` | 3 秒内回到 EditMode |
| PM-003 | 编译中 | 调用 `unity.playmode_start` | 返回忙碌或拒绝执行错误 |

## 4.5 鼠标输入模拟
| 用例ID | 前置条件 | 操作步骤 | 预期结果 |
| --- | --- | --- | --- |
| MS-001 | Scene 窗口可见 | `mouse_event(click,left,x,y)` | 返回 `executed=true` |
| MS-002 | Scene 窗口可见 | `down -> move -> up` | 每步成功，拖拽动作生效 |
| MS-003 | Scene 窗口可见 | `mouse_event(click,middle,x,y)` | 中键事件成功 |
| MS-004 | Scene 窗口可见 | `mouse_event(click,right,x,y)` | 右键事件成功 |
| MS-005 | 目标窗口不存在 | 发送 `mouse_event` | 返回窗口不可用错误 |

## 5. 验收脚本顺序（手工/半自动）
1. 启动 Orchestrator。
2. 打开 Unity，观察握手与心跳。
3. 执行 `CP-001` ~ `CP-004`。
4. 执行 `FX-001` ~ `FX-004`。
5. 执行 `PM-001` ~ `PM-003`。
6. 执行 `MS-001` ~ `MS-005`。
7. 记录结果并生成验收结论。

## 6. 结果记录模板

```text
[用例ID]：
- 时间：
- 平台：Windows / macOS
- 输入参数：
- 返回结果：
- 关键日志（requestId / commandId）：
- 判定：通过 / 失败
- 失败原因：
```

## 7. 通过门槛
1. 连接与协议：WS-001~WS-004 全通过。
2. 编译能力：CP-001~CP-004 全通过。
3. 修复闭环：FX-001 必须通过，FX-002/FX-003/FX-004 行为符合预期。
4. 交互能力：PM-001~PM-003 与 MS-001~MS-005 全通过。
5. 双平台要求：Windows 和 macOS 各至少执行 1 轮完整清单。

## 8. 失败处理规则
1. 任一 P0 用例失败，MVP 验收直接不通过。
2. 失败后必须补充 `requestId`、错误码、日志片段。
3. 修复后必须回归失败用例 + 相邻用例（同模块至少 3 条）。
