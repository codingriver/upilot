# M26-编辑器 E2E 自动化测试 — 具体开发方案

## 1. 文档目的

将「Agent 开发完编辑器工具后，可自动验证功能、数据、界面与日志」的能力**工程化**：规格可版本化、执行可重复、报告可归档，并与现有 MCP / Unity Bridge（`uitoolkit.*`、`screenshot.*`、Console、M17 测试运行等）对齐。

## 2. 目标与非目标

### 2.1 目标

| ID | 目标 | 验收要点 |
|----|------|----------|
| G1 | 用**机器可读规格**描述打开窗口、操作步骤、断言 | 同一份规格可被 Python 执行器与（可选）UTF 读取 |
| G2 | **端到端**通过 MCP 调用现有 Bridge 能力完成执行 | 不重复实现 UIToolkit 底层 |
| G3 | 断言分层：**UI 结构**（query/dump）、**数据**（反射/探针）、**日志**（Console 过滤）、**视觉**（截图可选） | 失败时附带 dump/截图/日志片段 |
| G4 | 输出**统一报告**（JSON）：每步状态、耗时、失败原因、附件引用 | CI / Agent 可解析 |

### 2.2 非目标（本阶段不做或单列子项目）

- 不替代 Unity Test Framework 的全部能力；UTF 作为 **Phase B** 可选接入。
- 不做通用「任意 Hook」暴露给 Agent（与 MonoHook 相关能力保持隔离，仅后续单独评估）。
- 视觉回归不以「像素级」为唯一标准；默认以 **query 断言** 为主。

## 3. 总体架构

```text
editor-e2e-spec.yaml (仓库内版本化)
        │
        ▼
┌───────────────────┐     WS      ┌─────────────────────────┐
│ Python 执行器      │ ────────►  │ unitypilot-editor       │
│ (MCP 工具或 CLI)   │  现有命令  │ (uitoolkit / screenshot / │
└─────────┬─────────┘            │  console / menu / …)     │
          │                       └─────────────────────────┘
          ▼
   report.json + artifacts/
   (截图 base64 或落盘路径、dump 摘要)
```

**原则**：Bridge 侧**尽量不新增**与业务耦合的命令；优先在 Python 侧实现「步骤编排 + 断言」。仅当缺能力时再在 C# 增加通用 primitive（例如 `wait.forEditorFrame`）。

## 4. 规格格式（v1）

路径建议：业务工程或本仓库 `UnityPilot/EditorTests/e2e/*.yaml`；本 monorepo 可放 `e2e-specs/examples/*.yaml`。

### 4.1 顶层字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `specVersion` | string | 固定 `1` |
| `name` | string | 用例名 |
| `timeoutS` | number | 整例超时（默认 300） |
| `defaults` | object | 默认 `targetWindow`、`waitMs` |
| `setup` | array | 前置步骤（菜单、编译等） |
| `steps` | array | 主步骤 |
| `teardown` | array | 可选清理 |

### 4.2 单步 `action` 枚举（v1）

与现有 MCP / Bridge 命令对齐，避免发明新动词：

| action | 主要参数 | 对应现有能力 |
|--------|----------|----------------|
| `menu` | `menuPath` | M15 `menu.execute`（或等价） |
| `wait` | `ms` 或 `until: editor_idle` | Python `asyncio.sleep`；若需主线程空闲再扩 Bridge |
| `uitoolkit.interact` | 与 Bridge payload 一致 | `uitoolkit.interact` |
| `uitoolkit.event` | 同上 | `uitoolkit.event` |
| `uitoolkit.setValue` | 同上 | `uitoolkit.setValue` |
| `uitoolkit.scroll` | 同上 | `uitoolkit.scroll` |
| `mouse` | 与现有 mouse_event 一致 | M05 |
| `keyboard` | 与现有 keyboard 一致 | 已有服务 |
| `screenshot.window` | `windowTitle` | `screenshot.editorWindow` |
| `console.clear` / `console.read` | 过滤、条数 | M07（需确认命令名与 tool_facade 一致） |

### 4.3 断言 `assert`（嵌在 step 后或独立 step）

| 类型 | 含义 | 实现 |
|------|------|------|
| `uitoolkit.query` | `name`/`className`/`textContains` 等匹配数量与字段 | 调 `uitoolkit.query`，在 Python 比较 |
| `uitoolkit.dump` | 子串或正则出现在序列化 dump 中 | 调 `uitoolkit.dump`（控制 `maxDepth`） |
| `console` | `maxErrors`、`mustContain`、`mustNotContain` | 读 Console 结果后比较 |
| `screenshot` | `optional: true` 时仅存档 | Base64 写入 `artifacts/` |
| `reflection.get` | （可选 Phase B）类型 + 静态字段路径 | `reflection` 相关工具 |

### 4.4 YAML 示例（节选）

```yaml
specVersion: "1"
name: "ExampleTool_Smoke"
timeoutS: 120
defaults:
  targetWindow: "My Custom Tool"
  waitMs: 200

setup:
  - action: menu
    menuPath: "Window/My Tool"

steps:
  - action: wait
    ms: 500
  - action: uitoolkit.interact
    targetWindow: "My Custom Tool"
    action: click
    elementName: "apply-button"
  - assert:
      type: uitoolkit.query
      targetWindow: "My Custom Tool"
      nameFilter: "status-label"
      expect:
        minMatches: 1
        textContains: "OK"

teardown:
  - action: menu
    menuPath: "Window/Close"   # 若存在
```

## 5. 分阶段开发计划

### Phase A — 规格 + Python 执行器 + MCP 入口（优先）

**目标**：不改或少改 C#，打通「读规格 → 调现有 facade → 出 report.json」。

| 序号 | 任务 | 产出 | 负责层 |
|------|------|------|--------|
| A1 | 定义 v1 JSON Schema（或 Pydantic 模型）与示例规格 2～3 个 | `e2e-specs/schema/spec-v1.json`、`examples/*.yaml` | Python |
| A2 | 实现 `EditorE2ERunner`：解析 YAML、逐步调用 `tool_facade` 同名方法 | `src/unitypilot_mcp/editor_e2e/` | Python |
| A3 | 统一异常与超时：每步计时、整例超时、失败收集 dump+截图 | `runner.py`、`report.py` | Python |
| A4 | 新增 MCP 工具 `unity.editor_e2e_run`（参数：`specPath` 或内联 YAML 字符串、`artifactDir`） | `mcp_stdio_server.py` + `tool_facade` | Python |
| A5 | 文档：如何在本仓库 / 用户工程放置规格、如何本地 `python -m unitypilot_mcp.editor_e2e ...` | 本节 + `README` 片段 | 文档 |
| A6 | 验收：对 `UnityPilotStatusWindow` 或现有示例窗跑通一条 smoke | 录屏或报告样例 | 联调 |

**依赖映射**：执行器内部只调用已有 `unity_*` facade 方法；若某 `action` 尚无 facade，在 A2 列清单并**最小补全** facade 封装（不新增 Bridge 命令除非必要）。

### Phase B — Unity Test Framework 对齐（可选）

**目标**：同一断言逻辑可在 **EditMode** 下无 MCP 运行，适合 CI。

| 序号 | 任务 | 产出 |
|------|------|------|
| B1 | 抽取 `IEditorE2EAssertions` 接口；Python 侧保留「远程」实现 | 设计文档 + 接口 |
| B2 | 在 `unitypilot-editor/Tests/Editor/` 增加 `EditorE2EDriverTests.cs`：用 `VisualElement` API 复现关键步骤 | UTF 用例 |
| B3 | MCP `unity.test.run`（M17）触发含上述程序集的测试 | 与现有 M17 联调 |

说明：B 阶段可与 A 并行，但 **A 先交付价值更快**。

### Phase C — Bridge 增强 primitive（按需）

仅在 Phase A 出现明确缺口时增加，例如：

| 能力 | Bridge 命令草案 | 用途 |
|------|-----------------|------|
| 主线程空闲等待 | `editor.wait`（延迟 N 帧或 ms） | 减少 UIToolkit 异步竞态 |
| 聚焦窗口 | 已有 Focus 路径则跳过 | — |
| 日志快照 token | `console.snapshot` 返回 id，后续 assert 对比增量 | 精确断言「本步之后」的日志 |

每增一项需更新 `docs/01-WS协议定义.md` 与 `docs/02-MCP工具接口定义.md`。

### Phase D — 视觉与基线（可选）

| 任务 | 说明 |
|------|------|
| D1 | `artifacts/` 下保存 PNG + manifest.json |
| D2 | 可选集成 `pillow` 或像素 diff（容差阈值） |
| D3 | 文档约定：视觉仅作辅助，门禁以 query 为主 |

## 7. MCP 工具设计要点

### 7.1 `unity.editor_e2e_run`

| 参数 | 类型 | 说明 |
|------|------|------|
| `spec_yaml` | string | 完整 YAML（适合 Agent 内联） |
| 或 `spec_path` | string | 工作区绝对/相对路径 |
| `artifact_dir` | string 可选 | 默认系统临时目录或 `.unitypilot/e2e-artifacts/<runId>` |
| `stop_on_first_failure` | bool | 默认 true |

**返回**：结构化 JSON：`passed`、`steps[]`、`failedStep`、`reportPath`、`artifacts`。

### 7.2 与现有工具关系

- **不删除**现有细粒度工具；`editor_e2e_run` 为**编排层**。
- Agent 日常仍可单步调用 `unity_uitoolkit_query` 调试；回归用 `editor_e2e_run`。

## 8. 报告格式（v1）

文件：`report.json`

```json
{
  "specName": "ExampleTool_Smoke",
  "passed": false,
  "startedAt": "ISO8601",
  "endedAt": "ISO8601",
  "steps": [
    {
      "index": 0,
      "action": "menu",
      "ok": true,
      "durationMs": 120,
      "detail": {}
    }
  ],
  "failure": {
    "stepIndex": 3,
    "message": "assert uitoolkit.query: expected minMatches 1, got 0",
    "attachments": {
      "uitoolkit.dump": "artifacts/03_dump.json",
      "screenshot": "artifacts/03_window.png"
    }
  }
}
```

## 9. 验收标准（M26）

1. 至少 **2 条**示例规格在连接 Unity 时 **全程通过**。
2. 故意断言失败时，报告含 **失败步骤索引 + 至少一种附件**（dump 或截图）。
3. `unity.editor_e2e_run` 在 Cursor MCP 中可调用并返回可读 JSON。
4. 文档说明与用户工程集成方式（规格放哪、路径如何传）。

## 10. 风险与缓解

| 风险 | 缓解 |
|------|------|
| UIToolkit 异步布局导致偶发失败 | `wait` 步骤 + Phase C `editor.wait`；规格中约定 `waitMs` |
| `screenshot.editorWindow` 仅部分平台 | 报告中标注 `skipped`，断言不依赖截图 |
| 规格与 Bridge 命令漂移 | `actions.py` 单点映射 + 集成测试 mock facade |

## 13. 实现落地（仓库现状）

| 组件 | 路径 / 说明 |
|------|-------------|
| YAML 示例 | `e2e-specs/examples/*.yaml` |
| Python 执行器 | `src/unitypilot_mcp/editor_e2e/`（`runner` / `actions` / `assertions` / `visual` / `report`） |
| MCP 工具 | `unity_editor_e2e_run(specPath, artifactDir?, stopOnFirstFailure?)` |
| 主线程延迟 | Bridge `editor.delay`；MCP `unity_editor_delay(delayMs)`；规格中 `action: wait` 默认走 `editor.delay` |
| UTF（Phase B） | `unitypilot-editor/Tests/Editor/` — `UnityPilot.Editor.Tests` |
| 依赖 | `PyYAML`、`pillow`（见 `pyproject.toml`） |

**视觉对比（Phase D）**：`assert.type: screenshot` 且提供 `baseline`（相对规格文件目录的 PNG 路径）时，用 Pillow 做逐像素容差对比（`tolerance` 默认 5）。
