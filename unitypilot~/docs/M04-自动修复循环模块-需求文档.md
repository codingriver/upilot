# M04-自动修复循环模块-需求文档

## 1. 模块职责
- 负责：
  - 执行“编译 -> 读取错误 -> 修复代码 -> 重新编译”闭环。
  - 控制循环上限与终止条件。
  - 输出每轮修复结果与最终状态。
- 不负责：
  - 不负责 Unity 命令传输协议实现。
  - 不负责鼠标和 PlayMode 控制。
  - 不负责业务功能正确性测试。
- 输入/输出：
  - 输入：编译错误列表、代码文件上下文。
  - 输出：修复后的文件变更、循环执行报告。

## 2. 数据模型
| 字段名 | 类型 | 语义 | 合法范围 | 默认值 |
| --- | --- | --- | --- | --- |
| loopId | string | 本次修复循环 ID | 非空 UUID | "" |
| maxIterations | int | 最大循环轮数 | `[1, 50]` | 20 |
| currentIteration | int | 当前轮次 | `[0, maxIterations]` | 0 |
| stopWhenNoError | bool | 无错误即停止 | `true/false` | true |
| lastCompileErrorCount | int | 最近一次错误数 | `[0, +∞)` | 0 |
| fixedFiles | list | 已修改文件列表 | 列表长度 `[0, +∞)` | [] |
| loopStatus | string | 循环状态 | `idle/running/success/failed/max_reached` | "idle" |

## 3. CRUD 操作
| 操作 | 入口 | 禁用条件 | Command类名 | Undo语义 |
| --- | --- | --- | --- | --- |
| 创建修复循环 | `auto_fix.start` | 已有循环 `running` | CreateFixLoopCmd | 不进 CommandHistory |
| 更新循环轮次 | 每轮开始时 | 循环不存在 | UpdateFixLoopIterationCmd | 不进 CommandHistory |
| 创建文件修复记录 | 每次写入文件后 | 文件路径为空 | AddFixedFileRecordCmd | 不进 CommandHistory |
| 更新循环状态 | 编译结束后 | 循环不存在 | UpdateFixLoopStatusCmd | 不进 CommandHistory |
| 删除循环上下文 | 循环结束后清理 | 循环不存在 | RemoveFixLoopContextCmd | 不进 CommandHistory |

## 4. 交互规格
- 触发事件：
  - 调用 `auto_fix.start` 启动循环。
- 状态变化：
  - `idle -> running -> success/failed/max_reached`。
- 数据提交时机：
  - 每轮编译结束后提交轮次结果。
  - 最终态在循环退出时提交。
- 取消/回退：
  - MVP 提供 `auto_fix.stop`，在下一轮开始前生效并将状态置为 `failed`。

## 5. 视觉规格
不涉及。

## 6. 校验规则
| 规则 | 检查时机 | 级别 | 提示文案 |
| --- | --- | --- | --- |
| `maxIterations` 必须在范围内 | 循环启动前 | Error | `最大轮次超出范围：{maxIterations}` |
| 编译错误为空时禁止继续修复 | 每轮修复前 | Info | `无编译错误，结束修复循环` |
| 同一轮最多修改 20 个文件 | 写文件前 | Warning | `本轮修改文件过多，已截断` |
| 单文件写入失败必须终止本轮 | 写文件时 | Error | `文件写入失败：{path}` |

## 7. 跨模块联动
| 模块 | 方向 | 说明 |
| --- | --- | --- |
| M03 编译与错误收集模块 | 被动接收 | 拉取每轮编译错误列表 |
| M01 编排服务模块 | 主动通知 | 上报每轮状态与最终结果 |
| M06 代码补丁执行模块 | 被动接收 | 应用修复补丁并返回写入结果 |

## 8. 技术实现要点
- 关键类与职责：
  - `AutoFixLoopService`：循环状态机。
  - `CompileErrorPlanner`：将错误映射到修复任务。
  - `FixLoopReporter`：输出结构化报告。
- 核心流程：
  - `start -> compile -> errors? -> patch -> compile -> ... -> stop`。
- 性能约束：
  - 单轮规划时间 `< 300ms`。
  - 20 轮内总耗时目标 `< 10min`。
  - 禁止在每轮读取整个仓库所有文件，仅读取错误关联文件。

## 9. 验收标准
1. [存在 2 处编译错误] -> [启动 `auto_fix.start`] -> [循环结束后编译错误为 0]。
2. [无编译错误] -> [启动循环] -> [1 轮内直接 `success`]。
3. [`maxIterations=1` 且错误未修复] -> [循环结束] -> [状态为 `max_reached`]。
4. [中途调用 `auto_fix.stop`] -> [下一轮开始前] -> [循环终止并返回 `failed`]。

## 10. 边界规范
- 空数据：错误列表为空时不生成补丁任务，直接成功退出。
- 单元素：仅 1 条错误时只处理该错误对应文件。
- 上下限临界值：`maxIterations=1` 与 `50` 均可正确终止。
- 异常数据恢复：某轮补丁失败后，记录失败原因并保留已成功轮次报告。

## 11. 周边可选功能
- P1：错误分组并行修复，预留 `fixGroupId` 字段。
- P2：跨轮次策略学习，预留 `iterationHints` 存储结构。
