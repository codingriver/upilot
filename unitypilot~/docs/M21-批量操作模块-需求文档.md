# M21-批量操作模块-需求文档

### 1. 模块职责

- 负责将多个独立的操作指令打包成单次请求发送给 Unity 编辑器。
- 负责控制多个任务按顺序或并行执行，并记录每个子任务的结果。
- 负责在批处理执行失败时，根据配置决定是否中止剩余任务或执行回滚操作。
- 不负责跨 WebSocket 通道的事务完整性一致性。
- 不负责长时间运行的后台任务异步解耦。
- 不负责处理具有复杂相互依赖关系的指令流。

### 2. 数据模型


| 字段名            | 类型     | 语义          | 合法范围                | 默认值        |
| -------------- | ------ | ----------- | ------------------- | ---------- |
| operations     | list   | 待执行的任务列表    | 对象数组 (tool/params)  | 空列表        |
| mode           | string | 执行模式        | sequential/parallel | sequential |
| stopOnError    | bool   | 出错时是否停止     | true/false          | true       |
| results        | list   | 任务执行结果详情    | JSON 对象数组           | 空列表        |
| totalTimeMs    | long   | 总执行耗时       | >= 0                | 0          |
| rollbackOnFail | bool   | 失败是否执行预定义回滚 | true/false          | false      |


### 3. CRUD 操作


| 操作   | 入口            | 禁用条件     | Command 类名         | Undo 语义           |
| ---- | ------------- | -------- | ------------------ | ----------------- |
| 批量执行 | batch.execute | 包含非法工具名称 | BatchExecuteCmd    | 不进 CommandHistory |
| 任务取消 | batch.cancel  | 已执行完毕    | BatchCancelCmd     | 不进 CommandHistory |
| 结果查询 | batch.results | 批处理未完成   | GetBatchResultsCmd | 不进 CommandHistory |


### 4. 交互规格

- **触发事件**：客户端发送 `batch.execute` 指令启动流水线。
- **状态变化**：批处理按策略遍历执行，状态依次转变为 `completed`。
- **数据提交时机**：所有子任务执行完毕后或因错误中断后汇总返回。
- **取消回退**：若支持回滚，失败时需逆序执行预置的撤回逻辑。

### 5. 视觉规格

不涉及。

### 6. 校验规则


| 规则      | 检查时机 | 级别      | 提示文案                |
| ------- | ---- | ------- | ------------------- |
| 指令白名单校验 | 启动前  | Error   | 批处理中包含未定义的工具指令      |
| 依赖循环校验  | 启动前  | Warning | 检测到可能的子任务逻辑死循环      |
| 超限校验    | 启动前  | Error   | 单次批处理指令数量超过上限 (100) |


### 7. 跨模块联动


| 模块             | 方向   | 说明                    |
| -------------- | ---- | --------------------- |
| M01编排服务        | 主动通知 | 请求批量任务的按序执行，减少网络往返    |
| 其他模块 (M07-M24) | 被动接收 | 接收来自 M21 的参数并同步执行原子操作 |


### 8. 技术实现要点

- **核心类**：`BatchProcessor` (执行器), `TaskQueue` (分发队列), `ResultAggregator` (结果汇聚器)。
- **执行流**：JSON 数据解析 -> 任务列表展开 -> 逐项分发给对应模块 -> 采集单项结果 -> 全局汇总回执。
- **性能约束**：并行模式下需限制并发线程数最大为 4；单次批处理耗时不得超过 60s。

### 9. 验收标准

- [前置条件] -> 定义包含 3 个 `unity.set_transform` 的列表 -> [操作] -> 顺序执行 -> [期望结果] -> 3 个物体位置均正确改变。
- [前置条件] -> `stopOnError` 为 true -> [操作] -> 第二个指令执行失败 -> [期望结果] -> 第三个指令不执行。
- [前置条件] -> 并行模式 -> [操作] -> 发送 5 个并发请求 -> [期望结果] -> 所有请求在更短总时间内完成。
- [前置条件] -> 输入任务超出 100 个 -> [操作] -> 发送请求 -> [期望结果] -> 触发超限拦截报错。

### 10. 边界规范

- **空数据**：传空任务列表时应返回空结果集。
- **单元素**：单任务批处理应与普通工具调用逻辑一致。
- **上下限临界值**：测试 100 个任务在复杂场景下的执行压力。
- **异常数据恢复**：若执行过程中断，需返回已完成任务的具体索引及结果。

### 11. 周边可选功能

- P1: 支持子任务之间的简单变量传递 (如 `{{last_result}}`)。
- P2: 增加批处理进度推送（WebSocket 增量回执）。注意：需预留对百分比通知的字段支持。
