# M03-编译与错误收集模块-需求文档

## 1. 模块职责
- 负责：
  - 提供 Unity 脚本编译触发入口。
  - 跟踪编译状态（开始、进行中、结束）。
  - 收集结构化编译错误并提供查询。
- 不负责：
  - 不负责错误自动修复策略。
  - 不负责运行时逻辑测试。
  - 不负责构建产物导出。
- 输入/输出：
  - 输入：`compile.request` 命令。
  - 输出：`compile.status` 事件、`compile.errors` 结果。

## 2. 数据模型
| 字段名 | 类型 | 语义 | 合法范围 | 默认值 |
| --- | --- | --- | --- | --- |
| compileRequestId | string | 当前编译请求 ID | 非空 UUID | "" |
| isCompiling | bool | 是否处于编译中 | `true/false` | false |
| compileStartedAt | long | 编译开始时间戳（ms） | `>= 0` | 0 |
| compileFinishedAt | long | 编译结束时间戳（ms） | `>= 0` | 0 |
| compileErrorCount | int | 本次编译错误数量 | `[0, +∞)` | 0 |
| compileWarningCount | int | 本次编译警告数量 | `[0, +∞)` | 0 |
| compileErrors | list | 错误列表 | 列表长度 `[0, +∞)` | [] |

## 3. CRUD 操作
| 操作 | 入口 | 禁用条件 | Command类名 | Undo语义 |
| --- | --- | --- | --- | --- |
| 创建编译任务 | `compile.request` | `isCompiling=true` | CreateCompileTaskCmd | 不进 CommandHistory |
| 更新编译状态 | 编译事件回调 | 未存在当前任务 | UpdateCompileStatusCmd | 不进 CommandHistory |
| 删除历史错误缓存 | 新一轮编译开始时 | 无 | ClearCompileErrorsCmd | 不进 CommandHistory |
| 创建错误记录 | 编译结束解析日志时 | 无编译任务 | AddCompileErrorRecordCmd | 不进 CommandHistory |
| 删除过期任务记录 | 保留窗口超限时 | 记录不存在 | RemoveExpiredCompileTaskCmd | 不进 CommandHistory |

## 4. 交互规格
- 触发事件：
  - 收到 `compile.request` 后调用 Unity 编译触发逻辑。
- 状态变化：
  - `idle -> compiling -> completed`。
- 数据提交时机：
  - 编译开始立即上报 `compile.status=started`。
  - 编译完成后上报 `compile.status=finished`，并更新错误列表。
- 取消/回退：
  - MVP 不支持中断编译；重复请求直接返回 `busy`。

## 5. 视觉规格
不涉及。

## 6. 校验规则
| 规则 | 检查时机 | 级别 | 提示文案 |
| --- | --- | --- | --- |
| 编译中禁止重复触发 | 收到 `compile.request` 时 | Error | `编译进行中，请稍后重试` |
| 错误记录必须包含 `file/line/message` | 写入错误列表前 | Error | `编译错误字段不完整` |
| 编译完成必须刷新状态 | 收到编译结束回调时 | Error | `编译结束状态未更新` |
| 请求 ID 必须匹配当前任务 | 回包时 | Warning | `请求 ID 不匹配：{compileRequestId}` |

| 模块 | 方向 | 说明 |

| M01 编排服务模块 | 被动接收 | 返回编译完成与错误计数 |

## 8. 技术实现要点
- 关键类与职责：
  - `CompileService`：触发编译与状态管理。
  - `CompileErrorCollector`：收集并结构化错误。
  - `CompileStateStore`：持久化当前编译状态。
- 核心流程：
  - `compile.request -> clear errors -> start compile -> on finish parse errors -> publish result`。
- 性能约束：
  - 错误解析耗时 `< 50ms`（100 条错误内）。
  - 状态查询接口响应 `< 10ms`。
  - 禁止每次查询都重新扫描 Console 全量日志。

## 9. 验收标准
1. [空闲状态] -> [调用 `compile.request`] -> [返回 accepted，状态变为 `compiling`]。
2. [代码存在语法错误] -> [编译完成] -> [`compile.errors` 返回 `file/line/message`]。
3. [连续触发两次编译] -> [第二次请求] -> [返回 `busy` 错误]。
4. [修复所有编译错误后再编译] -> [编译完成] -> [错误数量为 0]。

## 10. 边界规范
- 空数据：无错误时 `compileErrors=[]`，不得返回 `null`。
- 单元素：仅 1 条错误时，仍返回数组结构且长度为 1。
- 上下限临界值：错误数为 0 与 1000 时均可完整返回。
- 异常数据恢复：编译回调异常时，状态强制回退到 `completed` 并记录系统错误。
