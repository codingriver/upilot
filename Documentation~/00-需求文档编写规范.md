# 需求文档编写规范

> v5.0 | 2026-04-03
> 用途：规范 Agent 生成 `MXX-模块名-需求文档.md`，确保下游开发 Agent 可直接实现，不需要二次猜测。

---

## 0. 总则（必须遵守）

1. **需求文档是实现合同，不是讨论稿。** 只写可执行结论，不写分析过程、开放问题、过程性思考。
2. **必须按固定 11 章输出。** 章节编号、标题、顺序必须完全一致；不得缺章、并章、改名、重排。
3. **不得新增主章节。** 允许在章内使用 `###` 子标题，但不得出现 `## 12` 及之后的主章节。
4. **空章节也必须保留。** 若当前模块确实无内容，写 `无` 或 `不涉及`，不得省略。
5. **每条规则必须可映射到代码行为。** 至少能回答清楚：谁触发、在什么条件下、执行什么动作、产生什么结果、失败时怎么处理。
6. **所有范围、阈值、状态、时机、优先级必须显式写出。** 禁止使用“适当”“合理”“必要时”“尽量”等模糊词。
7. **优先使用表格。** 固定表头不得改名、删列、换序；自然语言只用于补充无法表格化的信息。
8. **可以引用其他文档，但不能只写“见某文档”。** 当前需求文档必须写出本模块最终结论，引用只能作为依据，不能代替结论。
9. **术语、字段名、命名方式必须与权威文档一致。** 不得自行发明与已有口径冲突的新名词。
10. **无法确定时必须采用最保守且不破坏数据的方案，并显式标注 `TODO(待确认)`。**
11. **需求文档必须与已有实现保持同步。** 若模块已有代码，文档中的类名、方法名、字段名、错误码必须与代码一致。若发现文档与代码不符，必须立即登记 TODO，不得以任一侧为准自行裁决。
12. **禁止凭空编造看似精确但未经验证的技术细节。** 类名、方法签名、协议参数等，要么从已有代码中提取，要么标注 `TODO(需原型验证)`。宁可留 TODO，不可虚构。

---

## 1. 固定章节结构（输出骨架）

生成需求文档时，必须严格使用以下 11 章：

```markdown
## 1. 模块职责
## 2. 数据模型
## 3. CRUD 操作
## 4. 交互规格
## 5. 视觉规格
## 6. 校验规则
## 7. 跨模块联动
## 8. 技术实现要点
## 9. 验收标准
## 10. 边界规范
## 11. 周边可选功能
```

---

## 2. 输出格式硬规则

### 2.1 文档级要求

- 文件名使用：`MXX-模块名-需求文档.md`
- 输出格式使用 Markdown
- 主章节只允许 `## 1` 到 `## 11`，必须使用二级标题（`##`），不得使用 `###`
- 章内允许 `###` 子标题，但不得破坏主章节顺序
- 表格、列表、伪代码均应直接服务于实现，不写装饰性内容

### 2.2 表格固定规则

下列表头必须原样使用：

- 数据模型：`字段名 | 类型 | 必填/可选 | 语义 | 合法范围 | 默认值`
- CRUD 操作：`操作 | 入口 | 禁用条件 | 实现标识 | Undo语义`
- 视觉规格：`状态 | 背景色 | 边框 | 文字颜色 | 备注`
- 输入校验：`规则 | 检查时机 | 级别 | 提示文案`
- 错误响应：`错误场景 | 错误码 | 错误消息模板 | 恢复行为`
- 跨模块联动：`模块 | 方向 | 说明 | 代码依赖点`

### 2.3 写作风格要求

- 优先用短句和确定句
- 一句话只表达一个结论
- 不写“等等”“相关内容”“类似处理”
- 不复制 GUI 文档原文；必须转写为可实现、可测试的需求

---

## 3. 各章节写作规则

### 3.1 模块职责

必须包含：

- 本模块负责什么
- 本模块不负责什么
- 输入对象 / 输出对象

要求：

- 责任边界必须写到可分工粒度，不能只写“负责管理 XXX”
- 若依赖其他模块提供入口或数据，必须写清依赖点

### 3.2 数据模型

作用：作为后续章节的唯一数据基线。

字段表必须使用：`字段名 | 类型 | 必填/可选 | 语义 | 合法范围 | 默认值`

强制规则：

- 字段名必须与实际数据结构一致
- `必填/可选` 列必须明确标注。请求/响应 DTO 的每个字段都必须标注 `必填` 或 `可选`，不得依赖隐式推断。
- 若字段可选，必须在 `合法范围` 列写明哨兵值（如 `-1 表示不修改`、`null 表示未设置`）及其语义。
- `合法范围` 必须写清是否包含端点
- 注释含 `范围 [a, b]`：在**写入处** Clamp，不在读取处 Clamp
- 注释含 `计算属性，不序列化`：使用 `get`-only，不写序列化字段
- `id` 字段：构造时初始化唯一值，如 `Guid.NewGuid().ToString()`
- 枚举一旦定义取值，逻辑分支必须覆盖全部枚举值
- 未写明可空时，默认不可空

### 3.3 CRUD 操作

操作表必须使用：`操作 | 入口 | 禁用条件 | 实现标识 | Undo语义`

强制规则：

- `入口` 必须写具体入口，如按钮、右键菜单、快捷键、API、自动流程
- `禁用条件` 同时作用于 UI 入口和 Execute 内二次校验
- `实现标识` 填写与操作对应的实际代码标识：Command 类名（如 `AddMarkerCmd`）、Service 方法名（如 `HandleCreate`）或 MCP tool 名（如 `unity_gameobject_create`）。必须与代码一致，不得虚构。
- `Undo语义` 必须明确 old/new 如何保存、如何恢复
- 级联删除必须写明删除顺序与 Undo 恢复顺序（逆序）
- 不可撤销操作必须显式标注：`不进 CommandHistory`
- 若某类 CRUD 在本模块不存在，写 `无`
- **CRUD 表必须覆盖该模块所有对外暴露的命令/API 端点。** 若模块已有实现代码，必须逐一比对已注册的 command handler / service method / MCP tool，确认 CRUD 表无遗漏。若某操作确实延后实现，必须在表中标注 `TODO(未实现)`，不得直接省略。

### 3.4 交互规格

必须写清：

- 触发事件
- 中间状态变化
- 数据提交时机
- 取消/回退行为

拖拽若未特别说明，默认状态机：

```text
PointerDown -> 记录 oldVal，进入拖拽视觉态
PointerMove -> 计算 newVal（先吸附再 Clamp），仅更新视觉
PointerUp   -> newVal != oldVal 时推 Command，并退出拖拽态
```

Command 推送时机默认值（除非本章明确覆盖）：

- 输入框：`BlurEvent` 或回车
- 拖拽：`PointerUpEvent`
- Toggle / ColorField：`ChangeEvent<T>`
- Esc：恢复原值，不推 Command

### 3.5 视觉规格

若模块涉及 UI，必须量化；若不涉及 UI，写 `不涉及`。

强制规则：

- 颜色仅允许 `#RRGGBB` 或 `rgba(r,g,b,a)`
- 尺寸仅允许 `px` 或 `flexGrow`
- 至少覆盖：`正常 / Hover / Selected / Disabled / 错误`
- 未声明的视觉属性默认继承正常态

状态表必须使用：`状态 | 背景色 | 边框 | 文字颜色 | 备注`

### 3.6 校验规则

#### 3.6.1 输入校验

校验表必须使用：`规则 | 检查时机 | 级别 | 提示文案`

级别语义固定：

- `Error`：阻断流程
- `Warning`：不阻断流程，但必须提示风险
- `Info`：仅提示信息，不改变流程

强制规则：

- 每条校验都要写检查时机
- 每条校验都要写提示文案
- 提示文案变量统一写作 `{变量名}`
- 若校验后有自动修复、自动 Clamp、自动回退，必须写明

#### 3.6.2 错误响应规格

错误响应表必须使用：`错误场景 | 错误码 | 错误消息模板 | 恢复行为`

强制规则：

- 每个 CRUD 操作的失败场景至少覆盖：参数非法、目标不存在、操作冲突
- 错误码必须全局唯一，命名规范为 `UPPER_SNAKE_CASE`
- 错误消息模板中的变量统一写作 `{变量名}`
- 恢复行为必须写明是：抛异常、返回错误对象、静默忽略、还是回退到上一状态
- 若模块已有实现代码，错误码必须与代码中已定义的保持一致
- 若模块不涉及错误响应（纯 UI 模块），写 `不涉及`

### 3.7 跨模块联动

联动表必须使用：`模块 | 方向 | 说明 | 代码依赖点`

方向只允许：

- `被动接收`
- `主动通知`

强制规则：

- 模块名优先写模块 ID，如 `M09 撤销重做`
- 说明必须写清触发点、传递内容或联动结果
- `代码依赖点` 必须写实际调用的类名.方法名（如 `UnityPilotComponentService.FindComponentByTypeAndIndex`）或事件/消息名。若模块为新设计，写 `(设计提案)`。
- 不得写“与其他模块联动”“同步刷新”等空话

### 3.8 技术实现要点

必须包含：

- 关键类与职责
- 核心流程（伪代码或实现骨架）
- 性能约束

强制规则：

- "建议"和"必须"要分开写；未标注的内容默认视为必须
- 性能约束必须可验证，如"禁止在高频事件中重建 UI 树"
- 若有明确禁止项，也应写在本章
- **关键类必须基于事实，禁止凭空编造。** 若模块已有实现代码，Agent 必须先搜索项目中已有的 Service/Manager/Handler 类，确认实际架构后再写。类名、方法签名必须与代码一致。
- 若模块为新设计（无现有代码），类名作为设计提案，必须标注 `(设计提案，实现时确认)`。
- 核心流程伪代码必须可直接映射到实际方法签名。若方法签名无法确认，标注 `TODO(待确认实际签名)`。
- 若某技术细节在需求阶段无法 100% 确定（如具体算法、协议细节、性能调优参数），必须标注 `TODO(需原型验证)` 并登记到 TODO.md。不得编造看似精确但未经验证的参数值。

### 3.9 验收标准

每条验收必须是**可执行步骤 + 可判定结果**，必须使用有序编号列表（`1.` `2.` `3.`），格式：

`1. [前置条件] -> [操作] -> [期望结果]`

强制规则：

- 每条验收都必须能判定通过或失败
- 至少覆盖主流程、关键异常、Undo/Redo（如适用）
- 禁止使用“功能正常”“表现正确”“体验良好”这类不可判定表述

### 3.10 边界规范

必须至少覆盖以下四类：

1. 空数据
2. 单元素
3. 上下限临界值
4. 异常数据恢复

强制规则：

- 每类都要写具体输入场景和期望行为
- 不得只写场景名，不写处理结果

### 3.11 周边可选功能

仅允许使用 `P1` / `P2`：

- `P1`：重要但非首版必须
- `P2`：低优先级

强制规则：

- 可选功能只能放在本章，不得混入前 10 章主流程
- 每项可选功能都要写清是否预留数据结构、接口或扩展点
- 若当前模块无可选功能，写 `无`

## 4. 信息来源优先级与缺失处理

### 4.1 生成需求文档时的信息来源优先级

发生冲突时，按以下顺序裁决（高 > 低）：

1. `00-Overview.md` 的“统一口径”
2. `01-数据模型总表.md`
3. `02-运行时数据协议.md`
4. 当前模块 `MXX-GUI布局文档.md`
5. `00-功能点列表.md`、`00-MVP功能列表.md`
6. `00-周边可选功能详情表.md`
7. 同类模块既有实现惯例

### 4.2 缺失处理默认值

- 未写 `null` 是否合法：默认**不合法**
- 未写默认值：`int/float=0`、`string=""`、`bool=false`、`List=空列表`
- 未写边界是否含端点：默认 `min/max` 均合法
- 未写操作提交时机：按本规范 §3.4 默认值处理
- 未写视觉状态：默认至少补齐 `正常 / Hover / Selected / Disabled / 错误`
- 仍无法判定时：采用最保守且不破坏数据的方案，并显式标注 `TODO(待确认)`

### 4.3 引用规则

- 可以引用权威文档作为依据
- 不得整段转抄外部文档
- 不得只写“详见 XXX 文档”
- 引用后必须在当前章节落成具体结论

## 5. 命名约定（文档中必须统一）

| 类型             | 规则                            | 示例                        |
| ---------------- | ------------------------------- | --------------------------- |
| Command 类       | `动词 + 名词 + Cmd`           | `AddMarkerCmd`            |
| 泛型属性 Command | `Set + 名词 + PropertyCmd<T>` | `SetMarkerPropertyCmd<T>` |
| Service 方法     | `Handle + 动词`               | `HandleCreate`            |
| MCP tool 名      | `snake_case`，`模块_动词_名词` | `unity_gameobject_create` |
| 错误码           | `UPPER_SNAKE_CASE`            | `OBJECT_NOT_FOUND`        |
| 数据类           | PascalCase                      | `SkillMarker`             |
| 枚举             | PascalCase                      | `MarkerType`              |
| 私有字段         | `_camelCase`                  | `_dragStartFrame`         |
| Command 描述     | 中文动宾短语                    | `"添加标记"`              |

## 6. 生成质量检查清单（Agent 自检后再输出）

输出前必须逐项自检：

- [ ] 11 个主章节齐全，标题使用 `##` 二级标题，顺序完全正确
- [ ] 没有新增 `##` 级主章节
- [ ] 空章节未省略，已写 `无` 或 `不涉及`
- [ ] 所有固定表格都使用了规定表头（含新增的 `必填/可选` 列、`实现标识` 列、`错误响应` 表、`代码依赖点` 列）
- [ ] 每条规则都可映射到"触发 / 条件 / 动作 / 结果 / 异常处理"
- [ ] 所有范围、阈值、状态、时机都已量化
- [ ] 所有输入校验都有检查时机、级别、提示文案
- [ ] 所有错误响应都有错误场景、错误码、错误消息模板、恢复行为
- [ ] 所有 CRUD 都有入口、禁用条件、实现标识、Undo语义；CRUD 表覆盖了所有对外暴露的命令/API
- [ ] 数据模型每个字段都标注了 `必填` 或 `可选`
- [ ] 验收标准使用有序编号列表，均可执行且可判定通过/失败
- [ ] 边界规范覆盖空、单、临界、异常恢复
- [ ] 可选功能只出现在第 11 章，且明确 P1/P2
- [ ] 未出现与权威文档冲突的术语、字段或结论
- [ ] 技术实现要点中的类名、方法名已通过搜索代码确认（或标注为设计提案）
- [ ] 跨模块联动的代码依赖点已写明实际类名.方法名
- [ ] 无法确认的内容均已标注 `TODO` 并登记到 `TODO.md`

## 7. 禁止项（出现即判不合格）

- 使用模糊词且无量化，如"适当""尽量""优化一下"
- 只写原则，不写触发条件、提交时机、处理结果
- 只写"见某文档"，不写当前模块结论
- 只写 happy path，不写边界与异常恢复
- 只写校验规则，不写级别和提示文案
- 只写视觉描述，不写颜色/尺寸/状态
- 将 P1/P2 混入前 10 章主流程
- 同一文档内前后结论冲突
- 表格缺列、改列名、换列顺序
- **凭空编造类名、方法名或架构设计**，未经代码搜索确认或未标注为设计提案
- **CRUD 表遗漏已实现的操作**，未与代码中的 command handler / service method / MCP tool 逐一比对
- **数据模型字段缺少必填/可选标注**，导致开发 Agent 无法判断参数是否可省略
- **错误处理只覆盖 happy path**，未定义错误码、错误消息模板和恢复行为

## 8. TODO 登记机制（强制执行）

### 8.1 TODO 文档位置

所有待确认、有歧义、代码与文档不一致的问题，统一登记到 `TODO.md`。

### 8.2 必须登记 TODO 的情形

以下任意一种情形出现，必须立即在 `TODO.md` 中新增一条记录，不得跳过：

| 触发情形             | 说明                                                               |
| -------------------- | ------------------------------------------------------------------ |
| 文档边界约束不全     | 字段合法范围、Clamp 上下界、边界含不含端点，任何一项未明确         |
| 文档内容缺失         | 某一交互路径、异常处理、视觉状态在需求文档中未描述                 |
| 文档存在歧义         | 同一文档内或跨文档出现可被多种方式解读的描述                       |
| 代码与文档不一致     | 实现代码的行为与需求文档描述存在偏差（无论是文档滞后还是实现偏差） |
| 无法 100% 确定的内容 | 写需求或写代码时，任何需要猜测或假设的结论                         |
| 实现复杂度超出需求精度 | 涉及状态机、并发、协议解析、性能优化等，需求文档只能写到接口契约级别，内部实现细节需原型验证后确认 |
| 跨技术栈边界         | 如 C#↔Python、UDP↔WebSocket 等跨运行时交互，序列化格式、错误传播、超时策略等细节需原型验证 |

### 8.3 TODO 条目格式

```markdown
### TODO-XXX：{一句话标题}

- **来源**：{文档名 §章节 / 代码文件:行号}
- **问题**：{具体描述：哪里不清楚、哪里冲突、哪里缺失}
- **当前处理**：{暂用的保守方案，或"阻塞待确认"}
- **需要决策**：{需要人工确认的具体问题，一问一句}
- **优先级**：{P0=阻塞开发 / P1=影响正确性 / P2=低风险}
```

### 8.4 强制规则

- TODO 编号连续递增，不得复用已用编号。
- `当前处理` 必须写出实际采用的保守方案，不得写"待定"或留空。
- `需要决策` 必须是可直接回答"是/否"或给出具体值的闭合问题，不得写开放式问题。
- TODO 解决后，在原条目末尾追加 `**已解决**：{结论}` 并更新对应需求文档，不得直接删除条目。
- 需求文档中对应位置的 `TODO(待确认)` 标注，必须在 TODO 解决后同步移除。

### 8.5 Agent 写作/实现时的自检

- 遇到"写不下去、只能猜"的点：先写保守实现，同时登记 TODO，**不得静默假设后继续**。
- 遇到需求文档与已有代码不符：**不得擅自以代码为准**，必须登记 TODO，注明两侧差异，由人工裁决。
- 同一问题不得重复登记；检查 `TODO.md` 是否已有相同条目，有则在原条目追加说明。
